1. Nombre del sistema

WayunaikLearn — Aplicación educativa para aprender Wayuunaiki (idioma wayúu).

2. Descripción del dominio

WayunaikLearn permite a usuarios registrarse, avanzar por niveles de aprendizaje, practicar con ejercicios y consultar un diccionario bilingüe (español ↔ Wayunaiki) con audios. Los administradores gestionan lecciones, ejercicios, categorías y audios para mantener actualizado el contenido y monitorear el progreso del aprendizaje.

3. Lenguaje ubicuo — términos clave

Términos principales del dominio (definición breve). Útiles para diseño de historias de usuario, APIs y documentación.

Usuario

Persona registrada que usa la app; posee progreso y niveles.

Administrador

Usuario con permisos para crear/editar lecciones, ejercicios y diccionario.

Lección

Conjunto temático de contenidos y ejercicios.

Ejercicio

Actividad evaluable dentro de una lección (enunciado, respuesta, dificultad).

Diccionario

Colección de entradas palabra (español ↔ Wayunaiki) con audios.

Entrada

Elemento del diccionario: palabra y su traducción, puede tener audio.

Audio

Archivo de sonido asociado a una entrada o lección.

Nivel

Agrupación progresiva de lecciones con requisitos y puntaje.

Progreso

Porcentaje, racha y logros del usuario.

Categoría

Clasificación temática de lecciones (ej.: Saludos, Familia).

Racha

Días consecutivos con actividad en la app.

Resultado

Evaluación de ejercicio (correcto / incorrecto).

Sesión

Uso continuo del usuario en la app.

Recurso

Elementos multimedia vinculados a lecciones o diccionario.

5. Arquitectura conceptual (capas y elementos)

División en tres capas: Dominio, Aplicación, Infraestructura. A continuación se describen entidades y casos de uso principales.

Capa de Dominio — Entidades y objetos de valor

  • Usuario (id, nombre, correo, contraseñaHash, progreso)

  • Lección (id, título, descripción, categoría, listaEjercicios)
  • Ejercicio (id, tipo, dificultad, enunciado, respuestaEsperada)
  • Nivel (id, nombre, requisitos, puntajeMinimo, logros)
  • Diccionario / Entrada (palabraEsp, palabraWayu, audioId)
  • Audio (id, ruta, duración)
  • Progreso (porcentaje, racha, logros)
  • ResultadoEjercicio (usuarioId, ejercicioId, correcto, fecha)

Capa de Aplicación — Casos de uso

  • RegistrarUsuario(datos)

6. Estándares del equipo (reglas)

  1. Nomenclatura: Clases en PascalCase; variables/funciones en camelCase; endpoints REST en plural (GET /usuarios).
  2. Ramas y commits: main, develop, feature/..., fix/..., release/.... Mensajes con prefijo: feat:, fix:.
  3. Revisión de código: Merge a develop requiere PR + al menos 1 aprobación y pruebas automáticas.
  4. Documentación mínima: README corto por endpoint/módulo; actualizar diagramas ER en migraciones.
  5. Prácticas prohibidas: No mergear a main/develop sin PR; evitar hardcodear credenciales.

7. Flujo de trabajo en equipo (simulación Git)

Descripción breve del flujo y proceso recomendado.

Diagrama ASCII del flujo

main  --------------------o--------------------> (production tags)
                         /
develop  ---o---o---o---/---o---o---o---o---     (integración continua)
           \   \   \   \   \   \   \   \
feature/ejercicios  feature/diccionario  feature/progreso ...
 \ (PR) -> Merge to develop after review
fix/ajuste-validacion -> PR -> develop
release/1.0 -> QA -> merge into main (tag v1.0) & deploy

Proceso recomendado: crear feature/ desde develop, desarrollar, PR, aprobar, merge a develop, pruebas, crear release y merge a main con tag.
